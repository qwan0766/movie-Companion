# 项目经验

## LLM vs 关键词匹配的选择权衡

### 背景

在将 5 个 Agent 的关键词匹配替换为 LLM 处理输入的过程中，对不同场景做了差异化处理。

### 已替换为 LLM（输入理解类）

| Agent | 原因 | 效果 |
|-------|------|------|
| IntentAgent | 意图分类需要语义理解，5 类边界模糊 | LLM 准确率显著高于关键词 |
| ChatAgent | 对话生成本身就是 LLM 的强项 | 回复自然，不再生硬 |
| KnowledgeAgent | 查询类型识别 + RAG 回答生成 | 能理解"介绍一下张毅"等自然表达 |
| PlanAgent | 时间/心情/类型混合提取，关键词难以覆盖 | LLM 能理解"周末想看轻松搞笑的" |

### 保持关键词匹配（工具层数据提取）

| 模块 | 原因 | 效果 |
|------|------|------|
| RetrievalAgent.parse_query | 结构化数据提取任务（年份/评分/类型/演员/地区） | 正则+关键词覆盖率 > 90%，毫秒级响应 |

### 关键判断：输入理解 vs 数据提取

Agent 的职责边界决定了是否使用 LLM：

- **Agent 层（输入理解）** → LLM
  - 理解用户"想干什么"（意图、查询类型、对话情绪）
  - 需要语义理解，模糊边界多

- **工具层（数据提取）** → 关键词/正则
  - 从文本中提取确定的结构化参数（年份、评分阈值、演员名）
  - 精确、确定、毫秒级

### LLM 替换关键词的代价

1. **延迟成本**：DeepSeek API 每次调用 5-10s，每增加一次 LLM 调用，用户多等 5-10s
2. **精确度下降**：正则提取"近三年" = 2023，LLM 可能含糊或算错
3. **确定性消失**：同一输入每次返回可能不同，测试和调试更困难
4. **成本增加**：每次 LLM 调用消耗 API 额度

### 总结原则

> Agent 的输入处理（理解用户意图）用 LLM，工具层的数据提取（结构化参数）用关键词。
>
> 前者是"理解什么"，后者是"提取什么"——两者性质不同，选择最优方案而非统一替换。

## Intent open-set upgrade

Current project remains a single-domain movie assistant, so a full domain layer is not introduced yet. The intent layer is upgraded from fixed labels to a thin open-set design:

- Keep core intents: `find_movie`, `ask_info`, `make_plan`, `chat`, `unknown`.
- Add control intents: `clarify`, `out_of_scope`.
- Add structured fields: `need_clarification`, `clarification_question`, `suggested_new_intent`.
- Route low-confidence results to `clarify` instead of forcing a business intent.
- Route unsupported tasks to `out_of_scope` so the assistant can state its capability boundary.

This keeps the graph simple while avoiding hard classification when the user input is vague or outside the Tencent Video assistant scope.

## Search and response presentation polish

When the user gives explicit structured constraints such as genre or content type, structured filtering should be treated as a hard preference before semantic recall. Semantic recall is still useful for broad queries, but for inputs such as `推荐科幻电影`, returning non-sci-fi or non-movie items hurts trust more than it helps diversity.

Response text shown in chat bubbles should prefer plain, stable line-based formatting over dense Markdown that may be compressed by HTML rendering. Detailed cards can still use rich UI components on the right panel.
